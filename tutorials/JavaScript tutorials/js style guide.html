JavaScript Style Guide And Coding Conventions

Selalu gunakan ketentuan persandian yang sama, untuk semua project JavaScript kamu.

Ketentuan Persandian JavaScript

Ketentuan persandian adalah garis panduan style untuk pemrograman, secara khas meliputi:

  • Aturan penamaan dan pendeklarasian pada Variabel dan Function.
  • Aturan pada white space, indentation, dan comment.
  • Pemrograman latihan dan prinsip.

Ketentuan persandian kualitas keamanan.

  • Meningkatkan kode agar lebih mudah dibaca
  • Membuat code pemeliharaan yang lebih mudah

Ketentuan persandian bisa didokumentasikan aturan pada kelompok untuk mengikuti, atau hanya menjadi latihan persandian individual kamu.

!

 Halaman ini mendeskripsikan ketentuan sandi JavaScript umu yang digunakan oleh w3school.
Kamu juga harus baca chapter selanjutnya "Best Practice" dan bagaimana untuk menghindari kode perangkap (pitfall).

Variable Names

Di Readbook kita menggunakan camelCase untuk pengidentifikasi nama (variabel dan fungsi).

Semua nama dimulai dengan huruf.

dibawah ini kamu akan menemukan diskusi yang lebih luas tentang aturan penamaan.

firstName = "John";
lastName = "Doe";

price = 19.90;
tax = 0.20;

fullPrice = price + (price * tax);

Spasi Disekitar Operator

Selalu taruh spasi disekitar operator (+-*/), dan setelah koma:

Contoh

var x = y + z;
var values = ["Volvo", "Saab", "Fiat"];

Code Indentations

Selalu gunakan 4 spasi untuk indentation blok kode:

Fungsi

function toCelsius(fahrenheit) {
    return (5 / 9) * (fahrenheit - 32);
}

!

 Jangan gunakan Tabs (tabulator) untuk indentation. Editor - editor yang berbeda menafsirkan tabs dengan cara yang berbeda.

Aturan Pernyataan

Aturan umum untuk pernyataan simple:

  • Selalu diakhir pernyataan sederhana tambahkan semicolon (;).

Fungsi

varvalues = ["Volvo", "Saab", "Fiat"];

var person = {
   
firstName: "John",
    lastName: "Doe",
    age: 50,
    eyeColor: "blue"
};

Aturan umum untuk pernyataan (campuran) kompleks:

  • Taruhlah pembukaan bracket (kurung) dibarisan pertama.
  • gunakanlah satu spasi sebelum pembukaan bracket.
  • Taruhlah penutupan bracket di garis baru tanpa ada spasi sebelumnya.

Functions

function toCelsius(fahrenheit) {
    return (5 / 9) * (fahrenheit - 32);
}

Loops

for (i = 0; i < 5; i++) {
   
x += i;
}

Loops

if (time < 20) {    
greeting = "Good day";
} else {    
greeting = "Good evening";
}

Aturan Object

Aturan umum untuk definisi Object:

  • Tempatkan pembukaan bracket pada baris yang sama sebagai nama object.
  • Gunakan colon (:) plus satu spasi diantara masing - masing properti dan nilai.
  • Gunakan tanda kutip disekitar nilai String (huruf), bukan disekitar nilai numeric (angka).
  • Jangan tambahkan koma pada properti atau nilai pada urutan terakhir.
  • Tempatkan penutupan bracket di baris baru, dan tanpa adanya spasi.
  • Selalu diakhir definisi object (dibelakang tanda kurung) tambahkan semicolon (;).

Complex Object

var person = {    
firstName: "John",
    lastName: "Doe",
    age: 50,
    eyeColor: "blue"
};

Object (short) bisa juga ditulis lebih singkat atau pendek, dengan satu baris dan penambahan satu spasi diantara properti seperti:

Short Object

var person = {firstName: "John", lastName: "Doe", age: 50, eyeColor: "blue"};

Panjang Baris < 80

Untuk lebih mudah dibaca, hidarilah panjang baris melebihi 80 karakter.
Jika pernyataan JavaScript tidak fit dalam satu baris, maka lebih baik kamu melanjutkannya di baris baru setelah operator atau koma.

Contoh

document.getElmentById("demo").innerHTML=
"Helo JavaScript";

Ketentuan Penamaan

Selalu gunakan ketentuan penamaan yang sama untuk semua kode kamu. Untuk contoh:

  • Nama variabel dan fungsi ditulis sebagai camelCase.
  • Variabel Global ditulis dalam UPPERCASE (kita tidak menggunakannya, tapi sangat umum digunakan).
  • Konstanta (seperti PI) ditulis dalam UPPERCASE.

Apakah sebaiknya kamu menggunakan hyp-hens, camelCase dan under_score pada nama - nama variabel ?
Ini adalah pertanyaan para programer yang seringkali ditanyakan. Jawabannya tergantung pada siapa kamu bertanya:
hyphens di HTML dan CSS:
Attribute HTML5 bisa dimulai dengan data- (data-quantity, data-price).
CSS menggunakan hyphens di properti-name (font-size)

!

 hypgens bisa disalahkan karena mirip seperti operator pengurangan (-), sebab itu hypgens tidak diperbolehkan di JavaScript.

underscore:

Para programer cenderung menggunakan underscore (data_the_birth) terutama pada SQL database.

pascalCase:

pascalCase sering diarahkan oleh C programer.

camelCase:

camelCase digunakan oleh JavaScript itu sendiri, oleh jQuery dan JavaScript libraries lainnya.

!

 Jangan memulai nama dengan $ sign, itu akan menaruh kamu dalam konflik dengan banyak nama - nama JavaScript library.

Memuat JavaScript pada HTML

Gunakan sintaksis sederhana untuk memuat external JavaScript (tipe JavaScript tidak diperlukan).

script src="script.js"

Pengaksesan Element HTML

Konsekuensi dari ketidak rapihan (untidy) HTML styles, mungkin hasil JavaScript akan error.
Disini dua pernyataan JavaScript yang memproduksi hasil yang berbeda:

em = document.getElementById("Demo")
em = document.getElementById("demo")

Jika memungkinkan gunakanlah ketentuan penamaan yang sama (seperti JavaScript) di HTML.
Kunjungilah panduan gaya tampilan HTML.

File Extensions

File HTML harus memiliki extension .html (bukan .htm)
File CSS harus memiliki extension .css
File JavaScript harus memiliki extension .js

Gunakan Lower Case Nama File

Sebagian banyak web server (Apache, Unix) adalah case sensitive terhadap nama file:
london.jpg tidak bisa diakses sebagai London.jpg.
Web server lainnya (window, IIS) tidak case sensitive:
london.jpg bisa diakses sebagai London.jpg atau london.jpg.
Jika menggunakan mencampurnya upper dan lower case, kamu harus konsisten yang sangat berbahaya.
Untuk menghindari Masalah masalah ini, lebih baik selalu gunakan lowercase untuk penamaan file (jika memungkinkan).

Performance (kinerja)

Ketentuan persandian tidak ditentukan oleh komputer, kebanyakan aturan memiliki sedikit impact (tubrukan / pengaruh yang kuat) pada pengeksekusian program.
Indentation dan spasi ekstra tidak signifikan pada small script.
Untuk kode dalam pengembangan, readability harus terarah, lebih besar berkas produksi (production script) harus diminimalkan.
